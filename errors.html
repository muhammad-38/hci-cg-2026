My "Identify and Fix" List

During the first month of class, I made these mistakes. Here is how I fixed them based on the Dix book and our lab rules.

HTML Practice: Learning from mistakes
The Error The Correction Why it's better (HCI Reason)
Using <b> to make a big heading. Using <h1> instead. Dix (p. 209) says structure helps navigation. Screen readers need headings to "read" the page map.
Forgetting alt text for images. Adding alt="Submit button". If the image doesn't load, the user gets stuck in a "Gulf of Evaluation" (Norman, 1988).
Putting <a> directly inside <ul>. Wrapping every link in an <li>. This is a strict Lab 01 rule. Lists must have list items for screen readers to work right.
Using the same id twice on one page. Giving every element a unique ID. Computers get confused if two things have the same ID. It breaks the interaction logic.
Writing "Click Here" for my links. Writing "View Week 1 PDF" instead. Dix mentions clarity. Descriptive links help the user's "Mental Processor" decide faster.
Not connecting a <label> to my input. Using for="id" to link them. This makes the label clickable, which helps with "Pointing" (P operator in KLM).
Using a table to fix my page layout. Only using tables for actual data. Tables are for comparison (like Norman vs GOMS), not for design.
Skipping <h2> and going straight to <h3>. Following h1 -> h2 -> h3. Broken hierarchy messes up the user's "Mental Model" of how the info is organized.
Missing the <!DOCTYPE html> at the top. Always starting with the Doctype. It tells the browser exactly how to render the pixels.
Putting color="red" inside the HTML tag. Using a <style> block at the top. Keeps the code clean. We should separate "content" from "computer graphics."

Books I double-checked:

  • Dix, A. et al. (2004). Human-Computer Interaction. (Ch 5 & 12).
  • Norman, D. (1988). The Design of Everyday Things.
